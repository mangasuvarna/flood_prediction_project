Rising Waters: A Machine Learning Approach to Flood Prediction
A flood risk prediction system trained on historical weather data (temperature, humidity, cloud cover, and seasonal rainfall) and served through a Flask web application, following the layered architecture below:

Browser -> Presentation Layer -> Flask App Layer -> ML Layer (model.pkl / scaler.pkl) -> Data Layer

Project Structure
flood_prediction_project/
├── app.py                 # Flask application (routes, prediction logic, history DB)
├── train_model.py         # Trains & compares 4 models, saves the best one
├── requirements.txt
├── data/
│   └── flood_dataset.xlsx
├── model/
│   ├── model.pkl           # Best trained model (generated by train_model.py)
│   ├── scaler.pkl          # Fitted StandardScaler (generated by train_model.py)
│   └── metadata.json       # Best model name, accuracy, comparison metrics
├── templates/
│   ├── base.html
│   ├── index.html           # Home dashboard
│   ├── form.html             # Input form (weather parameters)
│   ├── result.html          # Prediction result page
│   └── history.html         # Prediction history table
├── static/
│   └── css/style.css
└── history.db              # SQLite DB of past predictions (auto-created)
Setup
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
1. Train the model
python train_model.py
This loads data/flood_dataset.xlsx, trains Decision Tree, Random Forest, KNN, and XGBoost, evaluates each on a stratified 80/20 train/test split, and saves the best-performing model (by test accuracy) plus the fitted scaler to model/model.pkl and model/scaler.pkl.

2. Run the web app
python app.py
Visit http://127.0.0.1:5000 in your browser. Pages:

Home — dashboard with the active model and its metrics
New Prediction — enter weather parameters and get a flood/no-flood prediction
History — table of all past predictions, stored in a local SQLite database
Actual model results on this dataset
The dataset provided (flood_dataset.xlsx) has 115 rows with a notable class imbalance (99 no-flood vs. 16 flood events). Running train_model.py on this data produced:

Model	Accuracy	F1 Score
Decision Tree	95.65%	0.80
Random Forest	95.65%	0.80
KNN	86.96%	0.40
XGBoost	86.96%	0.00
Decision Tree was selected as the best model on this run (tied with Random Forest on accuracy, same F1). Note: with only 115 rows and 16 positive cases, a 20% test split is just 23 samples — accuracy numbers here have high variance and will shift depending on the random seed and split. XGBoost under-performed here specifically because it tends to need more data than this to avoid overfitting the majority class; it is not automatically the best choice on a small, imbalanced dataset like this one. If you need XGBoost specifically to be the deployed model, edit train_model.py to select it directly, or grow the dataset.

Retraining with more data
If you add more rows to data/flood_dataset.xlsx (same 10 feature columns + flood target), just rerun python train_model.py — it will re-evaluate all four models and overwrite model/model.pkl and model/scaler.pkl with whichever performs best.

Deployment
The app.py file uses Flask's built-in dev server for local testing. For production or cloud deployment (e.g. IBM Cloud, Render, Railway, etc.), run it behind a WSGI server such as gunicorn:

pip install gunicorn
gunicorn -w 2 -b 0.0.0.0:8080 app:app
